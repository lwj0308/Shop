**Q1：Spring Boot 的自动配置原理是什么？**

> 本项目关联：项目使用 Spring Boot 4.0.0，所有服务都是Spring Boot应用

**参考答案：**

Spring Boot 的核心是 `@SpringBootApplication` 注解，它包含三个核心注解：
1. `@SpringBootConfiguration`：本质就是 `@Configuration`，表示这是一个配置类
2. `@ComponentScan`：自动扫描当前包及其子包下的组件
3. `@EnableAutoConfiguration`：**核心，开启自动配置**

`@EnableAutoConfiguration` 的工作原理：
- 通过 `@Import(AutoConfigurationImportSelector.class)` 导入选择器
- 选择器会加载 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件（Spring Boot 3+）或 `META-INF/spring.factories`（Spring Boot 2）
- 文件中定义了所有自动配置类的全限定名
- 根据条件注解（`@ConditionalOnClass`、`@ConditionalOnMissingBean`、`@ConditionalOnProperty` 等）判断是否生效

**本项目中的例子**：
- 引入 `mybatis-plus-spring-boot4-starter` → 自动配置 MyBatis-Plus
- 引入 `spring-boot-starter-web` → 自动配置 Tomcat 和 Spring MVC
- 引入 `spring-boot-starter-data-redis` → 自动配置 RedisTemplate

**Q3：什么是微服务？微服务的优缺点？**

> 本项目关联：项目按业务拆成了 user/product/cart/order/payment/merchant/admin 7个服务

**参考答案：**

**微服务**是一种架构风格，将单体应用拆分成一组小型服务，每个服务：
- 独立部署、独立运行
- 围绕业务能力构建
- 通过轻量级通信机制（HTTP/RPC）互相调用
- 可以用不同语言/技术栈开发

**优点**：
1. **服务解耦**：各服务独立开发部署，修改一个服务不会影响其他服务（本项目中修改商品服务不影响订单服务）
2. **易于扩展**：热门服务可以单独扩容（如大促时只扩容商品和订单服务）
3. **技术灵活性**：不同服务可以用不同技术栈
4. **故障隔离**：一个服务挂了不会导致整个系统挂（但需要熔断机制保护）

**缺点**：
1. **复杂性高**：分布式系统本身更复杂（网络延迟、分布式事务、服务发现...）
2. **运维成本**：需要维护几十上百个服务的部署、监控、日志
3. **性能开销**：服务间调用有网络开销
4. **数据一致性**：跨服务数据一致性是难题

**Q4：@Transactional 注解的工作原理？哪些情况下会失效？**

> 本项目关联：订单服务取消订单、支付成功回调等使用了 @Transactional

**参考答案：**

**工作原理**：
Spring AOP 动态代理。在方法调用前后拦截：
- 方法执行前：开启事务（获取数据库连接，设置 `autoCommit=false`）
- 方法正常执行：提交事务
- 方法抛出异常（默认 RuntimeException）：回滚事务

**失效场景（高频面试题）**：
1. **方法不是 public**：Spring AOP 默认只代理 public 方法
2. **方法用 final/static 修饰**：final方法无法被子类重写，static方法属于类而非对象
3. **同一个类中方法调用**（最常见！）：`this.方法()` 不走代理
   ```java
   @Service
   public class OrderService {
       public void methodA() {
           this.methodB();  // ❌ this调用不走代理，@Transactional失效
       }
       @Transactional
       public void methodB() { ... }
   }
   ```
4. **数据库引擎不支持事务**：如 MySQL 的 MyISAM 引擎不支持事务（本项目用InnoDB，没问题）
5. **异常类型不对**：默认只回滚 RuntimeException 和 Error，手动 try-catch 吃掉异常也不回滚
6. **rollbackFor 设置错误**：需要明确指定 `@Transactional(rollbackFor = Exception.class)`

**本项目中的正确用法**（`OrderServiceImpl`）：
```java
@Override
@Transactional(rollbackFor = Exception.class)  // ✅ 正确：指定所有异常都回滚
public void cancelOrder(Long userId, Long orderId, OrderCancelDTO dto) {
    // ...业务逻辑
}
```

**Q5：Spring Bean 的生命周期？**

**参考答案：**

1. **实例化**：调用构造方法创建 Bean 对象
2. **属性赋值**：调用 setter 方法注入依赖（@Autowired字段注入）
3. **初始化前**：`@PostConstruct` 注解方法执行
4. **初始化**：调用 `InitializingBean.afterPropertiesSet()` 或 init-method
5. **初始化后**：AOP 代理增强（如果有 @Transactional、@Async 等）
6. **使用**：正常使用 Bean
7. **销毁**：容器关闭时调用 `@PreDestroy` 或 `DisposableBean.destroy()`

**简单记法**：实例化 → 属性赋值 → 初始化 → 使用 → 销毁

### 2. MySQL 相关

**Q6：MySQL 索引的底层数据结构？为什么用 B+ 树不用 B 树？**

> 本项目关联：项目每个表都建了索引，如 user(user_id)、order(order_no)

**参考答案：**

**MySQL InnoDB 引擎使用 B+ 树作为索引结构**

**B+ 树 vs B 树的区别**：

| 区别 | B树 | B+树 |
|-----|-----|------|
| 非叶子节点存数据 | ✓ 存完整数据 | ✗ 只存索引(key) |
| 叶子节点 | 独立存在 | 所有叶子节点用双向链表连接 |
| 查询效率 | 不稳定（可能在非叶子节点就找到） | 稳定（必须走到叶子节点） |
| 范围查询 | 需要遍历回根节点 | 直接通过叶子节点链表遍历，效率极高 |
| 单节点存储 | 数据占空间，树更矮但节点更少 | 只存key，节点更"瘦"，树更矮更宽 |

**为什么 B+ 树更适合做数据库索引？**：
1. **范围查询效率高**：叶子节点有链表，`WHERE id > 100 AND id < 200` 直接顺着链表找
2. **IO 次数更少**：每个节点能存更多索引值，树的高度更低（一般3-4层就能存千万级数据）
3. **查询稳定**：每次查询都必须走到叶子节点，性能稳定

**本项目例子**：`cart_item` 表的 `idx_user_id` 索引
```sql
-- B+树索引结构（简化）
-- 根节点 → 中间节点 → 叶子节点（含完整数据，链表连接）
CREATE INDEX idx_user_id ON cart_item(user_id);
```

**Q7：什么是聚集索引和非聚集索引（二级索引）？回表是什么？**

> 本项目关联：所有表都有主键（聚集索引）和业务字段索引（二级索引）

**参考答案：**

**聚集索引（Clustered Index）**：
- 索引和数据存放在一起（叶子节点存完整的行数据）
- 一个表只有一个聚集索引，默认是主键
- 数据物理上按索引顺序排列

**非聚集索引（Non-Clustered Index / 二级索引）**：
- 叶子节点只存主键值（不是完整数据）
- 一个表可以有多个二级索引

**回表（Table Access）**：
```sql
SELECT * FROM user WHERE phone = '138xxxxxxx';
-- 1. 通过 phone 索引（二级索引）找到主键 id
-- 2. 用主键 id 去聚集索引找完整数据 ← 这一步就是"回表"
```

**避免回表的方法**：
1. **覆盖索引**：查询的字段都在二级索引中，不需要回表
   ```sql
   CREATE INDEX idx_user_phone ON user(phone);
   SELECT id, phone FROM user WHERE phone = '138xxxxxxx';  -- ✅ 覆盖索引，不回表
   SELECT * FROM user WHERE phone = '138xxxxxxx';          -- ❌ 需要回表
   ```
2. **联合索引**：合理设计联合索引让常用查询命中覆盖索引

**Q8：什么是索引失效？常见的索引失效场景？**

> 本项目关联：需要审查项目中是否有写得不好的查询导致索引失效

**参考答案：**

**常见的索引失效场景**：

1. **LIKE 左模糊匹配**
   ```sql
   SELECT * FROM product WHERE name LIKE '%手机%';  -- ❌ 索引失效（%在前）
   SELECT * FROM product WHERE name LIKE '手机%';    -- ✅ 索引有效（%在后）
   ```
   *这也是为什么商品搜索要用 Elasticsearch 而不是 MySQL*

2. **对索引列使用函数/表达式**
   ```sql
   SELECT * FROM order WHERE YEAR(create_time) = 2024;  -- ❌
   SELECT * FROM order WHERE create_time >= '2024-01-01';  -- ✅
   ```

3. **隐式类型转换**
   ```sql
   -- phone 是 VARCHAR，但传了数字 13800000000，MySQL会转成数字比较
   SELECT * FROM user WHERE phone = 13800000000;  -- ❌ 隐式转换，索引失效
   SELECT * FROM user WHERE phone = '13800000000'; -- ✅
   ```

4. **OR 条件中有非索引字段**
   ```sql
   -- 只有 name 有索引，price 没有
   SELECT * FROM product WHERE name = '手机' OR price < 100;  -- ❌ 整体失效
   ```

5. **联合索引不满足最左前缀原则**
   ```sql
   -- 有联合索引 (category_id, brand_id, price)
   SELECT * FROM product WHERE category_id = 1 AND brand_id = 2;  -- ✅
   SELECT * FROM product WHERE brand_id = 2;                       -- ❌ 跳过左列
   SELECT * FROM product WHERE price < 100;                         -- ❌ 跳过左列
   SELECT * FROM product WHERE category_id = 1 AND price < 100;    -- ⚠️ 只用到category_id
   ```

6. **使用 !=、<> 操作符**（可能导致优化器放弃索引走全表扫描）

7. **IS NOT NULL**（IS NULL 可以命中，IS NOT NULL 可能失效）

**EXPLAIN 分析工具**：
用 `EXPLAIN SQL语句` 查看执行计划，重点看：
- `type` 列：system > const > eq_ref > ref > range > index > ALL（越靠左越好）
- `key` 列：实际使用的索引
- `rows` 列：预估扫描行数
- `Extra` 列：Using index（覆盖索引，好）/ Using where（需要回表）

**Q9：MySQL 事务的四大特性（ACID）？**

> 本项目关联：订单服务大量使用事务

**参考答案：**

| 特性 | 全称 | 说明 | MySQL 实现 |
|-----|------|------|-----------|
| **A**tomicity | 原子性 | 事务中的操作要么全部成功，要么全部失败 | **Undo Log（回滚日志）** |
| **C**onsistency | 一致性 | 事务前后数据保持一致（如余额不能为负） | 应用层面保证 + 其他特性共同保证 |
| **I**solation | 隔离性 | 多个事务互不干扰 | **MVCC（多版本并发控制）+ 锁** |
| **D**urability | 持久性 | 事务提交后数据永久保存 | **Redo Log（重做日志）** |

**Q10：MySQL 事务的四大隔离级别？脏读/不可重复读/幻读是什么？**

> 本项目关联：分布式事务中也涉及隔离级别的概念

**参考答案：**

**四个隔离级别（从低到高）**：

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | MySQL默认 |
|---------|-----|----------|-----|----------|
| Read Uncommitted（读未提交） | ✓ | ✓ | ✓ | ❌ |
| Read Committed（读已提交） | ✗ | ✓ | ✓ | ❌ |
| **Repeatable Read（可重复读）** | ✗ | ✗ | ✓ | ✅ 默认 |
| Serializable（串行化） | ✗ | ✗ | ✗ | ❌ |

**三种问题解释**：
1. **脏读**：事务A读到事务B修改了但还没提交的数据（B回滚了，A读的就是"脏"数据）
2. **不可重复读**：事务A两次读同一行数据，但数据被事务B修改并提交了，两次读的结果不一样（针对 UPDATE/DELETE）
3. **幻读**：事务A两次按相同条件查询，事务B新增/删除了符合条件的数据，结果集大小不一样（针对 INSERT）

**MySQL 默认隔离级别是 RR（Repeatable Read）**，通过 MVCC + Next-Key Lock 基本解决了幻读问题。

**Q11：MVCC 是什么？实现原理？**

**参考答案：**

**MVCC（Multi-Version Concurrency Control，多版本并发控制）**：

核心思想：读写不冲突，读的时候用快照，写的时候加行锁。

**实现原理**（基于 InnoDB）：

每行数据都有 **3 个隐藏字段**：
1. `DB_TRX_ID`（6字节）：最后一次修改/插入该行的事务ID
2. `DB_ROLL_PTR`（7字节）：回滚指针，指向 undo log 中的上一个版本
3. `DB_ROW_ID`（6字节）：隐藏主键（如果表没有主键）

配合 **undo log（回滚日志）**：
- 每次修改数据时，都会在 undo log 中记录旧版本，形成一条版本链
- 读的时候，根据 Read View 判断哪个版本对当前事务可见

**Read View（读视图）** 包含：
- `m_ids`：当前活跃的事务ID列表（已启动但未提交）
- `min_trx_id`：最小的活跃事务ID
- `max_trx_id`：下一个要分配的事务ID
- `creator_trx_id`：创建这个 Read View 的事务ID

**可见性判断规则**：
- 版本的 `trx_id < min_trx_id` → 可见（已经提交了）
- 版本的 `trx_id >= max_trx_id` → 不可见（这个版本开始于当前事务启动之后）
- `trx_id` 在 `m_ids` 中 → 不可见（这个事务还没提交）
- `trx_id` 不在 `m_ids` 中 → 可见（这个事务已经提交了）

**Q12：MySQL 的 redo log、undo log、binlog 有什么区别？**

**参考答案：**

| 日志 | 产生时机 | 作用 | 级别 |
|-----|---------|------|------|
| **redo log** | InnoDB引擎产生 | **崩溃恢复**（WAL：Write-Ahead Logging，先写日志再写磁盘） | 物理日志（页级修改） |
| **undo log** | InnoDB引擎产生 | **回滚事务** + MVCC版本链 | 逻辑日志（反向操作） |
| **binlog** | Server层产生 | **数据备份** + **主从同步** | 逻辑日志（SQL语句或行变更） |

**两阶段提交（保证 redo log 和 binlog 一致）**：
```
阶段1：Prepare → 写 redo log，标记为 prepare 状态
阶段2：Commit → 写 binlog → 写 redo log commit 标记
```

### 3. Redis 相关

**Q13：Redis 为什么这么快？**

> 本项目关联：库存扣减、分布式锁、缓存都用到了 Redis

**参考答案：**

1. **纯内存操作**：所有数据存在内存中，读写速度极快（10万+ QPS）
2. **单线程模型**（Redis 6 之前）：避免了多线程上下文切换和锁的开销
   - 注意：单线程指的是**网络IO和键值对读写**由一个线程完成
   - 持久化、集群同步等由其他线程完成
3. **IO多路复用**：使用 epoll（Linux）/ kqueue（Mac）实现非阻塞IO，一个线程处理大量连接
4. **高效的数据结构**：SDS、跳表、压缩列表等，操作时间复杂度低

**Q15：Redis 分布式锁的实现？如何保证可靠性？**

> 本项目关联：订单服务使用 Redis 分布式锁防止同一用户并发下单

**参考答案：**

**基础版本**：
```bash
SET lock:order:user123 "value" NX EX 30
# NX = if Not eXists（不存在才设置，获取锁）
# EX = 过期时间（秒），防止死锁
```
释放锁：`DEL lock:order:user123`

**问题**：释放锁时可能误删别人的锁（A超时了，B获取锁，A恢复后把B的锁删了）

**改进版本 - 释放锁时校验值**：
```bash
# 获取锁时设置唯一标识
SET lock:order:user123 "requestId-A" NX EX 30

# 释放锁时检查是不是自己加的锁（Lua脚本保证原子性）
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
else
    return 0
end
```

**RedLock 红锁算法**（多Redis节点，更可靠但复杂）：
1. 向 N 个独立的 Redis 节点依次加锁
2. 超过半数 (N/2+1) 节点加锁成功，且耗时小于锁过期时间 → 认为获取锁成功
3. 释放锁时向所有节点发送释放命令

**本项目的分布式锁实现**（简化版，在 `OrderServiceImpl`）：
```java
Boolean locked = stringRedisTemplate.opsForValue()
    .setIfAbsent("order:create:" + userId, "locked", 30, TimeUnit.SECONDS);
// 注意：本项目实现比较简单，没有做value校验，生产环境需要完善
```

**Q16：Redis 缓存穿透、缓存击穿、缓存雪崩的区别和解决方案？**

> 本项目关联：商品数据可能面临这些问题，目前项目未做完整处理

**参考答案：**

| 问题 | 描述 | 解决方案 |
|-----|------|---------|
| **缓存穿透** | 查询**不存在的数据**，缓存和数据库都没有，每次请求都打到数据库 | 1. 缓存空值（布隆过滤器更高级）<br>2. 参数校验，拦截非法请求 |
| **缓存击穿** | **某个热点key**刚好过期，大量并发请求同时涌入数据库 | 1. 热点key永不过期<br>2. 加互斥锁（分布式锁）<br>3. 随机化过期时间 |
| **缓存雪崩** | **大量key在同一时间集中过期**，或 Redis 实例宕机 | 1. 过期时间加随机值<br>2. 多级缓存（本地缓存+Redis）<br>3. Redis 集群/哨兵高可用<br>4. 服务降级/熔断 |

---

**Q17：Redis 持久化机制 RDB 和 AOF 的区别？**

**参考答案：**

| 特性 | RDB（快照） | AOF（日志） |
|-----|------------|------------|
| **原理** | 定时将内存数据的**二进制快照**保存到磁盘 | 以日志形式**追加记录每个写命令** |
| **数据完整性** | 较差（可能丢失最后一次快照后的数据） | 较好（最多丢失1秒数据） |
| **文件大小** | 小（压缩的二进制） | 大（文本日志，需要重写压缩） |
| **恢复速度** | 快（直接加载快照） | 慢（重放所有命令） |
| **默认开启** | ✅ 是（`save 900 1` 等规则） | ❌ 否，需配置 `appendonly yes` |

**推荐方案**：Redis 4.0+ 支持 **RDB-AOF 混合持久化**
- 用 RDB 做全量备份（快速恢复）
- 用 AOF 做增量备份（保证数据不丢失）

**Q18：Redis + Lua 脚本的原子性？本项目怎么用的？**

> 本项目关联：`StockServiceImpl.java` 中使用 Lua 脚本扣减库存

**参考答案：**

**Redis 执行 Lua 脚本是原子的**：
- Redis 是单线程处理命令的
- 整个 Lua 脚本会作为一个"大命令"一次性执行
- 执行过程中不会被其他命令打断 → 天然原子性

**本项目的 Lua 脚本应用（扣减库存）**：

```lua
-- 脚本：检查库存 → 检查是否扣减过 → 扣减库存 → 记录订单号
if redis.call('SISMEMBER', KEYS[2], ARGV[2]) == 1 then
    return 0  -- 已经扣减过了（幂等）
end
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock < tonumber(ARGV[1]) then
    return 0  -- 库存不足
end
redis.call('DECRBY', KEYS[1], ARGV[1])  -- 扣减库存
redis.call('SADD', KEYS[2], ARGV[2])    -- 记录订单号
return 1  -- 成功
```

**为什么不用多次 Redis 命令？**：
```java
// ❌ 非原子，并发情况下可能超卖
int stock = redis.get(skuId);       // 步骤1
if (stock >= quantity) {
    redis.decrby(skuId, quantity);  // 步骤2 → 这两步之间可能被其他请求修改
}

// ✅ Lua脚本，原子操作
redis.execute(script, keys, args);
```

**Java 代码调用 Lua 脚本**（`StockServiceImpl`）：
```java
DefaultRedisScript<Long> script = new DefaultRedisScript<>(LUA_SCRIPT, Long.class);
Long result = stringRedisTemplate.execute(script, List.of(stockKey, deductRecordKey), 
                                          String.valueOf(quantity), orderNo);
```

### 4. RocketMQ / 消息队列相关

**Q19：为什么要用消息队列？消息队列的优缺点？**

> 本项目关联：支付成功通知订单服务、订单超时自动取消都使用 RocketMQ

**参考答案：**

**三大核心作用**：
1. **异步解耦**：支付服务处理完后发个消息就返回，不用等订单服务处理
2. **削峰填谷**：大促时大量请求涌入，消息队列做缓冲，下游服务按能力消费
3. **数据分发**：一个事件可以被多个消费者处理（如支付成功 → 更新订单+发短信+加积分）

**优点**：
- 降低系统耦合度
- 提高系统响应速度（异步）
- 提高系统吞吐量（削峰）
- 保证消息可靠传递

**缺点**：
- 系统复杂度增加（需要维护MQ集群）
- 消息可能重复或丢失 → 需要保证幂等和可靠性
- 消息延迟 → 不适合强实时场景

**本项目中的使用**：
- `topic_order_timeout`：下单时发送30分钟延时消息，超时未支付自动取消订单
- `topic_payment_callback`：支付成功后通知订单服务更新状态

---

**Q20：消息队列如何保证消息不丢失？**

**参考答案：**

从 **生产者 → MQ → 消费者** 三个环节分析：

1. **生产者端**：
   - 使用同步发送（`syncSend`）：等待 MQ 确认收到
   - 发送失败重试机制
   - 事务消息（RocketMQ支持）

2. **MQ 端（Broker）**：
   - 消息持久化到磁盘（RocketMQ 默认异步刷盘，可配置同步刷盘）
   - 主从同步（Master 和 Slave 都保存消息）
   - 避免 Broker 宕机导致消息丢失

3. **消费者端**：
   - 手动 ACK 模式（`CONSUME_SUCCESS` 或 `RECONSUME_LATER`）
   - 先处理业务逻辑，成功后再 ACK
   - 消费失败时返回重试状态，MQ 会重新投递

**Q21：如何处理消息重复消费？**

> 本项目关联：支付成功消息可能重复投递，订单状态更新需要保证幂等

**参考答案：**

**核心思想：保证消费端幂等性**（同一消息消费多次和消费一次结果一样）

**常见幂等方案**：

1. **唯一键 + 数据库唯一索引**
   - 消费消息后，将消息ID插入带唯一索引的表
   - 重复消费会触发唯一键冲突，直接跳过

2. **Redis Set/原子操作**（本项目库存扣减使用此思路）
   - 用 Redis 的 `SETNX` 或 Lua 脚本判断消息是否已处理
   ```
   if (redis.setnx("processed:" + msgId, "1")) {
       // 第一次消费，处理业务
       redis.expire("processed:" + msgId, 3600);
   } else {
       // 已处理过，跳过
   }
   ```

3. **状态机判断**（本项目订单状态更新使用）
   - 订单有明确的状态流转：待付款 → 已付款 → 待发货 → 运输中 → 已完成
   - 重复收到"支付成功"消息时：
     ```java
     // 幂等校验：检查订单当前状态
     if (order.getStatus() == OrderStatusEnum.PAID.getCode()) {
         log.info("订单已支付，忽略重复回调");  // 已支付，直接返回成功
         return;
     }
     // 乐观锁更新（where status = 待付款），并发情况下只有一个能成功
     ```

4. **版本号/乐观锁**

**本项目中的幂等处理**（`OrderServiceImpl.paySuccess`）：
- 先判断订单状态是否已经是"已支付"
- 是则直接返回（幂等），否则用乐观锁条件更新状态

**Q22：RocketMQ 的延时消息是怎么实现的？**

> 本项目关联：下单时发送30分钟延时消息用于超时取消

**参考答案：**

**RocketMQ 延时消息原理**：
- Broker 收到延时消息后，**不立即投递**给消费者
- 将消息存到 `SCHEDULE_TOPIC_XXXX` 这个系统 Topic 中
- 有专门的定时任务定期扫描这些消息，到期后再投递到原始 Topic
- 消费者从原始 Topic 正常消费

**使用方式**（本项目）：
```java
// 延时等级16 = 30分钟
// RocketMQ 固定18个等级：1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h
rocketMQTemplate.syncSend("topic_order_timeout", message, 3000, 16);
```

**注意**：RocketMQ 4.x 只支持固定延时等级，不支持任意时间。RocketMQ 5.1+ 支持任意时间延时。

### 5. Elasticsearch 相关

**Q23：为什么要用 Elasticsearch 做商品搜索？为什么不用 MySQL？**

> 本项目关联：商品搜索使用 Elasticsearch 8.15.0

**参考答案：**

**MySQL 的问题**：
1. `LIKE '%keyword%'` **不能命中索引**，全表扫描极慢（千万级数据秒级响应是奢望）
2. 不支持**中文分词**（搜"手机"搜不到"智能手机"）
3. 不支持**相关性排序**（无法按匹配度打分排序）
4. 不支持**聚合统计**（如搜索结果中各品牌/分类的数量）
5. 不支持**高亮显示**

**Elasticsearch 的优势**：
1. **倒排索引**：以"词"为单位建立索引，"词 → 文档列表"，搜索极快
2. **IK 中文分词**：支持中文分词，能理解自然语言
3. **相关性评分（TF-IDF / BM25）**：按匹配度给文档打分排序
4. **分布式**：天然支持分布式，可横向扩展处理海量数据
5. **丰富的查询 DSL**：全文检索、范围查询、聚合、高亮、建议

**本项目中的使用**（`ProductSearchServiceImpl`）：
- `multiMatch`：在 name 和 subtitle 字段中搜索关键词
- `term`：按 categoryId、brandId 精确筛选
- `range`：按价格区间筛选
- `highlight`：关键词高亮
- `aggregation`：品牌/分类聚合统计
- `suggest`：搜索建议

**Q24：什么是倒排索引？和正排索引的区别？**

**参考答案：**

假设我们有3篇文章：
- Doc1：iPhone 15 Pro 手机
- Doc2：华为 Mate 60 Pro 手机
- Doc3：小米 14 手机壳

**正排索引**（文档 → 内容，MySQL用的）：
```
Doc1 → [iPhone, 15, Pro, 手机]
Doc2 → [华为, Mate, 60, Pro, 手机]
Doc3 → [小米, 14, 手机壳]
```
查询"手机"时：逐篇文章扫描，检查是否包含"手机" → O(n)，慢

**倒排索引**（词 → 文档列表，ES用的）：
```
iPhone → [Doc1]
15 → [Doc1]
Pro → [Doc1, Doc2]
手机 → [Doc1, Doc2]
华为 → [Doc2]
Mate → [Doc2]
60 → [Doc2]
小米 → [Doc3]
14 → [Doc3]
手机壳 → [Doc3]
```
查询"手机"时：直接查索引，命中 [Doc1, Doc2] → O(1)，快

**Q25：ES 数据同步的策略？本项目怎么实现的？**

> 本项目关联：商品数据存储在 MySQL，需要同步到 ES

**参考答案：**

**常见同步方案**：

1. **双写**（简单但不可靠）：写 MySQL 后立即写 ES
   - 问题：双写可能不一致（第二个写失败）、有耦合

2. **定时任务（T+1）**（简单）：定时拉取 MySQL 数据全量/增量同步
   - 问题：数据延迟大（小时级）

3. **Canal / Maxwell**（异步）：监听 MySQL binlog，解析变更并同步到 ES
   - 原理：伪装成 MySQL 从库，订阅 binlog 事件
   - 优点：实时性好、无业务侵入
   - 阿里出品，配合 RocketMQ 可实现高可用

4. **消息队列异步同步**（本项目使用思路）：
   - 商品创建/更新后 → 发 MQ 消息 → 消费者同步到 ES

**本项目的实现**（`ProductSyncConsumer`）：
- 商品变更后发送消息到 `product_sync_topic`
- 消费者从 MySQL 查询最新商品数据 → 写入 ES
- 搜索时**实时价格**从 MySQL 合并（价格和库存是强实时数据，不缓存到ES）

### 6. 分布式事务相关

**Q26：什么是分布式事务？CAP 定理？BASE 理论？**

> 本项目关联：订单服务使用 Seata 处理跨服务分布式事务

**参考答案：**

**分布式事务**：一次事务操作涉及多个服务/多个数据库，需要保证跨服务的数据一致性

**CAP 定理**：分布式系统中，**Consistency（一致性）、Availability（可用性）、Partition tolerance（分区容错性）**三者不可兼得，只能满足两个：
- **CP**：Zookeeper（一致性优先，分区时不可用）
- **AP**：Eureka/Nacos（可用性优先，数据可能短暂不一致）
- P 是必须的（网络分区不可避免），所以实际是在 C 和 A 之间权衡

**BASE 理论**（对 CAP 的妥协，互联网项目常用）：
- **Basically Available**（基本可用）：允许部分故障
- **Soft state**（软状态）：允许中间状态
- **Eventually consistent**（最终一致性）：最终数据会一致

**分布式事务解决方案对比**：

| 方案 | 原理 | 一致性 | 性能 | 适用场景 |
|-----|------|--------|------|---------|
| **2PC/3PC**（两阶段提交） | 准备阶段 + 提交阶段 | 强一致 | 低 | 传统银行等金融场景 |
| **TCC**（Try-Confirm-Cancel） | 业务代码实现三个阶段接口 | 最终一致 | 中 | 对业务改造要求高 |
| **Seata AT**（本项目用） | 自动生成回滚日志（undo_log） | 最终一致 | 高 | 业务侵入小，推荐 |
| **Saga** | 长事务分解，失败时反向补偿 | 最终一致 | 高 | 长链路业务 |
| **本地消息表/MQ** | 发消息 + 消费端幂等处理 | 最终一致 | 高 | 支付通知等 |

---

**Q27：Seata AT 模式的原理？**

> 本项目关联：订单服务的 `@GlobalTransactional` 使用 Seata AT 模式

**参考答案：**

**Seata AT 模式核心：自动生成回滚日志**

**流程**（本项目：创建订单 → 扣减库存）：

1. **TM（事务管理器，发起方）**：订单服务启动全局事务
   ```java
   @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
   public OrderDetailVO createOrder(Long userId, OrderCreateDTO dto) { ... }
   ```

2. **第一阶段 - 执行本地事务**：
   - 每个 RM（资源管理器，参与方）执行业务 SQL
   - **自动**：解析 SQL → 生成前镜像（修改前的数据）→ 执行业务 SQL → 生 成后镜像 → 写入 undo_log 表
   - 同时获取本地行锁（通过 SELECT ... FOR UPDATE）
   - 提交本地事务，释放连接（**注意：本地事务已经提交了**）

3. **第二阶段 - 提交或回滚**：
   - **如果所有分支都成功**：TC 通知所有 RM 提交 → 删除 undo_log，释放全局锁
   - **如果任一分支失败**：TC 通知所有 RM 回滚 → 根据 undo_log 反向执行 SQL，恢复数据 → 删除 undo_log，释放全局锁

**关键表**：本项目中 `shop_order` 数据库包含 `undo_log` 表

**优点**：业务代码零侵入，几乎不需要修改原有的本地事务代码
**缺点**：依赖数据库，高并发场景下全局锁可能成为瓶颈

### 7. Spring Cloud Gateway 相关

**Q28：Gateway 的工作原理？和 Zuul 的区别？**

> 本项目关联：shop-gateway 使用 Spring Cloud Gateway

**参考答案：**

**Gateway 核心组件**：
1. **Route（路由）**：请求转发的规则，由 ID、目标 URI、断言、过滤器组成
2. **Predicate（断言）**：匹配 HTTP 请求的内容（路径、Header、参数等）
3. **Filter（过滤器）**：请求/响应的预处理/后处理（鉴权、限流、日志、Header修改等）

**请求处理流程**：
```
客户端请求 → Gateway Handler Mapping（匹配路由） → Gateway Web Handler
     ↓
Filter Chain（全局过滤器 + 路由过滤器，按order排序）
     ↓
代理到目标微服务 → 返回响应 → 反向通过过滤器链 → 返回给客户端
```

**Gateway vs Zuul 的区别**：

| 特性 | Zuul 1.x | Spring Cloud Gateway |
|-----|---------|---------------------|
| 底层实现 | 同步阻塞的 Servlet | 响应式 WebFlux + Netty |
| 性能 | 较低（线程模型） | 较高（Reactor，非阻塞） |
| 长连接支持 | 较差 | 好（WebFlux天然支持） |
| 动态路由 | 需要额外开发 | 内置支持 |

**Q29：本项目的鉴权流程是怎么设计的？**

> 本项目关联：`AuthGlobalFilter` 实现网关鉴权

**参考答案：**

**鉴权流程**：
```
请求进来 → 判断是否在白名单 → 是 → 直接放行（如登录、商品浏览）
              ↓ 否
         检查 Token（从 Header 取）
              ↓ 无效/缺失
         返回 401 未登录
              ↓ 有效
         清除敏感 Header（防伪造）
         写入 userId 和角色信息到 Header
         ↓
         转发给下游微服务（user/product/order...）
         下游服务从 Header 中获取 userId 识别用户
```

**核心要点**：
1. **白名单机制**：登录、商品浏览等接口不需要鉴权
2. **Token 自动续期**：Token 快过期时自动续期（Sa-Token 实现）
3. **Header 防伪造**：网关删除客户端传入的 `X-User-Id` 等敏感 Header，由网关统一写入
4. **服务间信任**：下游服务信任网关写入的 Header（注意：服务间调用也要鉴权）

**潜在优化点**：
- 目前实现较简单，可增加限流、异常用户识别等

### 9. JVM 相关

**Q31：JVM 内存结构？**

**参考答案：**

JDK 8+ 的内存结构：
```
JVM 内存
├── 堆（Heap） → 所有线程共享
│   ├── 年轻代（Young）
│   │   ├── Eden区（8/10）
│   │   ├── Survivor 0（1/10）
│   │   └── Survivor 1（1/10）
│   └── 老年代（Old）→ 默认占堆2/3
├── 元空间（Metaspace）→ 方法区实现，使用本地内存
│   ├── 类信息
│   ├── 常量池
│   ├── 静态变量
│   └── JIT编译后的代码
├── 虚拟机栈（VM Stack）→ 每个线程私有
│   └── 栈帧（局部变量表、操作数栈、方法返回地址...）
├── 本地方法栈（Native Method Stack）
└── 程序计数器（PC Register）→ 当前线程执行的字节码行号
```

**GC 针对的区域**：主要是堆（年轻代 + 老年代）

**Q32：JVM 垃圾回收算法？常见的垃圾回收器？**

**参考答案：**

**GC 算法**：

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|-----|------|------|------|---------|
| **标记-清除** | 标记可达对象 → 清除未标记 | 简单 | 内存碎片、效率不高 | 老年代基础算法 |
| **复制算法** | 内存分两块，只使用一块，存活对象复制到另一块 | 无碎片、速度快 | 浪费内存、存活对象多则效率低 | 年轻代（Eden→Survivor） |
| **标记-整理** | 标记 → 整理（存活对象向一端移动）→ 清除边界外 | 无碎片 | 需要移动对象，成本高 | 老年代 |
| **分代收集** | 不同代用不同算法 | 综合最优 | - | 主流方案 |

**常见垃圾回收器**：

| 回收器 | 年代 | 特点 | JDK版本 |
|-------|------|------|---------|
| Serial | 新/老 | 单线程，STW时间长 | 早期 |
| Parallel Scavenge | 新生代 | 多线程并行，吞吐量优先 | JDK 8 默认 |
| CMS | 老年代 | 低延迟，并发标记，STW短 | JDK 8 可选 |
| **G1** | 全代 | 区域划分、可预测停顿时间、大内存友好 | JDK 9+ 默认 |
| **ZGC** | 全代 | 极低延迟（ms级）、TB级堆、染色指针 | JDK 15+ 生产可用 |

### 10. Java 基础

**Q33：HashMap 的实现原理？为什么线程不安全？**

**参考答案：**

**JDK 1.8+ HashMap 结构**：数组 + 链表 + 红黑树

**核心字段**：
- `table`：Node[] 数组（哈希桶）
- `size`：元素个数
- `loadFactor`：负载因子，默认 0.75
- `threshold`：扩容阈值 = 容量 × 负载因子

**插入流程**：
1. 计算 key 的 hash 值（`(h = key.hashCode()) ^ (h >>> 16)`）
2. 计算数组下标：`(n-1) & hash`（n是数组长度，保证范围）
3. 如果数组位置为空 → 直接插入
4. 如果已有元素（哈希冲突）：
   - key 相同 → 覆盖 value
   - 是红黑树节点 → 红黑树插入
   - 是链表 → 尾插法；插入后链表长度 ≥ 8 且数组长度 ≥ 64 → 转红黑树
5. 插入后 size > threshold → 扩容（容量 × 2，所有元素重新哈希）

**为什么线程不安全？**：
1. **并发 put 可能丢数据**：两个 put 计算同一个下标，可能互相覆盖
2. **并发扩容可能形成环形链表**（JDK 1.7 问题，头插法）→ 死循环
3. **put 和 get 并发**：get 可能读到不一致的数据

**线程安全的 Map**：
- `ConcurrentHashMap`：分段锁 / CAS + synchronized（JDK 1.8+），性能好
- `HashTable`：全表加 `synchronized`，性能差，不推荐
- `Collections.synchronizedMap`：包装类，全方法加锁

---

**Q34：ConcurrentHashMap 的实现原理？**

**参考答案：**

**JDK 1.8+ 实现**：

1. **数据结构**：Node[] 数组 + 链表 + 红黑树（和 HashMap 类似）
2. **锁机制**：CAS（数组位置）+ synchronized（每个桶的首节点）
   - 空桶时：用 CAS 插入（无锁，性能高）
   - 已有元素时：用 synchronized 锁住桶的首节点（锁粒度比分段锁更小）
3. **扩容**：多线程协作扩容，每个线程负责一部分桶
4. **get 操作无需加锁**：Node 的 val 和 next 用 volatile 修饰，保证可见性

**和 Hashtable 的区别**：
- Hashtable：每个方法 synchronized，锁整个表，并发度=1
- ConcurrentHashMap：锁每个桶的首节点，并发度=桶数量

**Q35：synchronized 和 ReentrantLock 的区别？**

**参考答案：**

| 特性 | synchronized | ReentrantLock |
|-----|-------------|---------------|
| 实现方式 | JVM层面，关键字 | API层面，Java类 |
| 锁释放 | 自动（代码块执行完或异常） | 必须手动 unlock() |
| 可重入 | ✓ 是 | ✓ 是 |
| 公平锁 | ✗ 非公平 | ✓ 可配置（默认非公平，性能好） |
| 可中断 | ✗ 不能 | ✓ lockInterruptibly() |
| 超时获取 | ✗ 不能 | ✓ tryLock(timeout) |
| 条件变量 | ✗ wait/notify笨拙 | ✓ Condition 多条件 |
| 性能 | 低竞争时性能好（偏向锁→轻量级→重量级膨胀） | 高竞争时更可控 |
| 死锁检测 | - | 可通过编程检测 |

**使用建议**：优先用 synchronized，需要高级功能时用 ReentrantLock。

**Q36：volatile 关键字的作用？和 synchronized 的区别？**

**参考答案：**

**volatile 的两个作用**：
1. **保证可见性**：volatile 变量的修改会立即刷新到主内存，其他线程能看到最新值
2. **禁止指令重排序**：通过内存屏障阻止编译器和 CPU 对 volatile 变量前后的指令重排

**经典应用**：
- 双重检查锁定（DCL）单例模式
```java
public class Singleton {
    private static volatile Singleton instance;  // 必须加volatile！
    private Singleton() {}
    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();  // 可能被重排序
                }
            }
        }
        return instance;
    }
}
```
- 不加 volatile 的问题：`new Singleton()` 可能被重排序为：分配内存 → 设置instance引用 → 初始化对象，导致其他线程拿到"半初始化"的对象

**volatile vs synchronized**：
- volatile 只能修饰变量，synchronized 修饰方法/代码块
- volatile 不保证原子性（如 `i++` 仍是三步），synchronized 保证原子性
- volatile 无锁，不阻塞，性能更好

### 场景1：高并发扣减库存

**Q：电商大促时，1000个用户同时抢购100件商品，如何保证库存不超卖？**

**参考答案（结合本项目实现）**：

**方案1：数据库乐观锁（简单但性能一般）**
```sql
UPDATE product_sku 
SET stock = stock - #{quantity}, version = version + 1 
WHERE id = #{skuId} AND stock >= #{quantity} AND version = #{version};
```
- 优点：简单，不依赖中间件
- 缺点：高并发下大量请求会失败（version 冲突），用户体验差，数据库压力大

**方案2：Redis + Lua 脚本（本项目使用 ⭐推荐）**

原理：Redis 单线程 + Lua 脚本原子执行
```lua
-- 脚本内容（在 StockServiceImpl 中）
-- 1. 检查该订单号是否已经扣减过（防止重复请求）
-- 2. 检查库存是否足够
-- 3. 扣减库存 + 记录订单号
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil or stock < tonumber(ARGV[1]) then
    return 0  -- 库存不足
end
if redis.call('SISMEMBER', KEYS[2], ARGV[2]) == 1 then
    return 0  -- 已经扣减过
end
redis.call('DECRBY', KEYS[1], ARGV[1])
redis.call('SADD', KEYS[2], ARGV[2])
return 1
```

**方案3：Redis 预扣库存 + 数据库异步扣减**
- Redis 存可用库存，扣减在 Redis 完成（毫秒级）
- 同步创建订单数据库记录
- 支付成功后再真正扣减数据库库存（或定期对账）

**方案4：分库分表 + 库存分片**
- 将同一个商品的库存分到不同库的不同分片
- 每个分片独立扣减
- 进一步提高吞吐量

**本项目选择方案2的原因**：
- 性能：Redis 内存操作，QPS 可达 10万+
- 简单：Lua 脚本保证原子性，业务逻辑清晰
- 幂等：订单号去重，防止重复扣减

**完整流程**：
```
用户下单 → 获取分布式锁（防止同一用户重复下单）
   ↓
调用商品服务（Feign）→ Redis Lua脚本扣减库存（Seata分布式事务）
   ↓ 成功
创建订单 → 发送延迟消息（30分钟超时自动取消）
   ↓
删除购物车 → 释放分布式锁 → 返回成功
   ↓ 超时未支付
RocketMQ延迟消息消费 → 取消订单 → 回滚库存（Lua脚本+1）
```

### 场景3：订单超时自动取消

**Q：用户下单后30分钟未支付，如何自动取消订单并回滚库存？**

**参考答案（本项目使用 RocketMQ 延时消息）**

**方案1：定时任务轮询数据库（简单但性能差）**
```
定时任务（每分钟执行）：
SELECT * FROM order_info WHERE status = 0 AND create_time < NOW() - 30min
→ 批量取消 + 回滚库存
```
- 缺点：轮询数据库有压力、取消不精确（可能延迟1分钟）

**方案2：Redis 过期键通知（中等方案）**
- 下单时设置 Redis key `order:timeout:{orderNo}`，过期时间30分钟
- 配置 Redis `notify-keyspace-events Ex`，监听过期事件
- 收到过期事件后取消订单
- 缺点：Redis 过期不精确（被动删除）、单机模式下可靠性一般

**方案3：RocketMQ 延时消息（本项目使用 ⭐推荐）**

```
用户下单 → 发送延时消息（等级16 = 30分钟）
   ↓
消息存到 topic_order_timeout（不会立即被消费）
   ↓ 30分钟后
RocketMQ 消费者收到消息
   ↓
查询订单状态：
   ├── 已支付 → 什么也不做，记录日志
   └── 待付款 → 调用 cancelOrder()：
                1. 更新订单状态为"已取消"（乐观锁更新）
                2. 回滚库存：Redis Lua脚本 +1
                3. 记录状态变更日志
```

**代码示例**（`OrderServiceImpl`）：
```java
// 发送延时消息
rocketMQTemplate.syncSend(
    "topic_order_timeout",
    MessageBuilder.withPayload(orderNo).build(),
    3000,  // 发送超时
    16     // 延时等级16 = 30分钟
);

// 消费者（简化）
@RocketMQMessageListener(topic = "topic_order_timeout", consumerGroup = "order_timeout_group")
public class OrderTimeoutConsumer implements RocketMQListener<String> {
    public void onMessage(String orderNo) {
        orderService.autoCancelOrder(orderNo);  // 内部检查状态 + 乐观锁更新
    }
}
```

**方案4：定时任务 + 延时消息（双保险，更可靠）**
- 主要用延时消息（实时性好）
- 定时任务每天扫一遍，作为补偿机制兜底
- 防止消息丢失导致订单永远处于"待付款"

### 场景4：分布式事务 - 跨服务数据一致性

**Q：创建订单需要同时扣减库存和扣减用户余额，分别在不同服务/数据库，如何保证一致性？**

**参考答案**：

**业务场景**：
```
shop-order 服务（订单数据库）
   ↓ Feign 调用
shop-product 服务（商品数据库）→ 扣减库存
   ↓ Feign 调用
shop-user 服务（用户数据库）→ 扣减余额
```

**方案1：Seata AT 模式（本项目使用）**

```java
@GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
public OrderDetailVO createOrder(Long userId, OrderCreateDTO dto) {
    // 1. 本地：创建订单（shop-order数据库）
    orderInfoMapper.insert(orderInfo);
    orderItemMapper.insertBatch(orderItems);
    
    // 2. 远程：扣减库存（shop-product服务）
    productFeignClient.deductStock(skuId, quantity);  // ✅ Seata自动参与事务
    
    // 3. 远程：扣减余额（shop-user服务）
    userFeignClient.deductBalance(userId, amount);     // ✅ Seata自动参与事务
    
    // 任何一步抛出异常 → 全局事务回滚（所有数据库操作自动撤销）
}
```

**原理**：每个参与服务需要 `undo_log` 表，Seata 自动记录修改前的数据快照，回滚时自动执行反向 SQL。

**方案2：TCC 模式（Try-Confirm-Cancel）**
- Try：冻结库存 + 冻结余额
- Confirm：真正扣减（如果全部 Try 成功）
- Cancel：解冻（如果任何一个 Try 失败）
- 优点：不依赖数据库锁，并发能力强
- 缺点：业务侵入大，每个接口需要写三个版本

**方案3：Saga 模式（长事务）**
- 按顺序执行服务，如果某一步失败，按相反顺序执行补偿操作
- 适合业务链路长、可补偿的场景

**方案4：最终一致性（消息队列 + 本地消息表）**
- 订单服务创建订单 → 发"扣库存"消息到 MQ
- 商品服务消费消息 → 扣库存
- 如果扣库存失败 → 发"取消订单"消息
- 订单服务消费 → 取消订单
- 优点：性能高、解耦好
- 缺点：实现复杂（需要保证消息可靠性、幂等）

**对比**：

| 方案 | 一致性 | 业务侵入 | 性能 | 推荐场景 |
|-----|--------|---------|------|---------|
| Seata AT | 最终一致 | 低 | 中 | 业务简单，追求开发效率（本项目） |
| TCC | 最终一致 | 高 | 高 | 对性能要求极高 |
| Saga | 最终一致 | 中 | 高 | 长链路业务 |
| 消息队列 | 最终一致 | 中 | 高 | 异步场景、非强一致性要求 |

### 场景6：缓存一致性问题

**Q：先更数据库还是先删缓存？如何保证数据库和缓存的数据一致性？**

**参考答案**：

**经典 Cache Aside Pattern**：

**读**：先读缓存 → 命中返回 → 未命中读数据库 → 写入缓存 → 返回

**写**（两种方案）：

| 方案 | 步骤 | 问题 |
|-----|------|------|
| **先删缓存，再更数据库** | 1. delete cache<br>2. update DB | ✅ 简单<br>❌ 并发读可能缓存脏数据（删缓存后，读请求先到，查旧数据写回缓存） |
| **先更数据库，再删缓存** | 1. update DB<br>2. delete cache | ✅ 脏数据概率低<br>❌ 第二步删除失败会不一致 |

**推荐方案：先更数据库，再删缓存 + 延迟双删**

```java
public void updateProduct(Long productId, ProductUpdateDTO dto) {
    // 1. 先更新数据库
    productMapper.updateById(product);
    
    // 2. 再删除缓存
    stringRedisTemplate.delete("product:" + productId);
    
    // 3. 延迟双删（可选，应对极端并发场景）
    // 睡眠一小段时间后再删一次，确保并发读不会把旧数据写回缓存
    new Thread(() -> {
        try {
            Thread.sleep(500);  // 睡眠时间 > 读数据库+写缓存的时间
            stringRedisTemplate.delete("product:" + productId);
        } catch (Exception e) {}
    }).start();
}
```

**终极方案：Canal 监听 binlog 异步删除缓存**
- 数据库变更 → Canal 监听 binlog → 发消息到 MQ → 消费者删除缓存
- 完全解耦，可靠性高（即使应用崩溃，缓存也能最终一致）

**本项目目前的做法**：
- 价格/库存强实时数据：**不走缓存，直接查 MySQL**（搜索结果合并时实时查询）
- 商品基本信息：可缓存，依赖商品变更同步

### 场景7：接口幂等性设计

**Q：用户网络卡顿重复点击"提交订单"，怎么防止生成多个订单？**

**参考答案**：

**幂等性**：同一请求执行一次和执行 N 次结果相同

**方案1：前端控制（第一道防线）**
- 点击后按钮置灰 + 显示loading，防止重复点击
- *注意*：只能防普通用户，防不住恶意脚本或网络重放

**方案2：Token 机制（前端+后端配合 ⭐本项目使用）**

```
步骤：
1. 用户进入下单页面 → 前端请求后端获取幂等Token
   GET /api/order/idempotent-token
   ↓
   后端：String token = UUID();
        redis.set("order:idempotent:" + token, "1", 5, MINUTES);
        返回 token
   
2. 前端提交订单时带上 Token
   POST /api/order/submit  + Header: X-Idempotent-Token: xxxxx

3. 后端用 Lua 脚本原子检查并删除 Token
   if redis.call('GETDEL', 'order:idempotent:' + token) == '1' then
       // 获取成功，继续创建订单（这个Token已被删除，重复请求会失败）
   else
       // Token不存在，返回"请勿重复提交订单"
   end
```

**方案3：数据库唯一约束**
- 订单号 + 用户ID 建唯一索引（但订单号应该是唯一的，这不适用）
- 更适合：同一用户同一商品一天内只能领一张优惠券

**方案4：分布式锁 + Redis Set 去重**
- 下单时获取分布式锁 `order:create:{userId}`
- 将 skuId+userId 组合作为 key 存入 Set 做去重

**方案5：状态机判断（订单状态流转）**
- 支付回调等场景：订单已有明确状态（待付款→已付款）
- 重复收到"支付成功"消息时：先检查状态，已支付则直接返回

**本项目的幂等策略总结**：
| 场景 | 方案 |
|-----|------|
| 创建订单 | 前端Token + 分布式锁 + Lua脚本去重 |
| 扣减库存 | Redis Set 记录订单号（SISMEMBER判断） |
| 支付回调 | 订单状态机判断 + 乐观锁更新 |
| 取消订单 | 乐观锁条件更新 |

### 场景8：服务雪崩与熔断降级

**Q：商品服务挂了，订单服务调用它会超时，最终导致整个系统雪崩，怎么防止？**

**参考答案**：

**服务雪崩**：服务A调用服务B，服务B故障导致A大量请求阻塞等待，A也被拖垮，连锁反应

**解决方案**：

**1. 超时控制**（本项目需要优化）
- Feign 设置合理的超时时间（连接超时 + 读取超时）
```yaml
feign:
  client:
    config:
      default:
        connectTimeout: 2000   # 连接超时 2秒
        readTimeout: 5000       # 读取超时 5秒
```

**2. 重试机制（谨慎使用）**
- 读操作可以重试（如查询商品信息）
- 写操作谨慎重试（可能重复写）→ 配合幂等设计
- 指数退避：第一次失败后等待100ms，第二次200ms，第三次400ms

**3. 熔断降级（核心方案 ⭐）**

推荐使用 **Sentinel**（Spring Cloud Alibaba）：

```
正常状态（Closed）：请求正常通过
   ↓ 异常比例/超时比例超过阈值（如20%）
熔断状态（Open）：5分钟内所有请求直接返回降级响应
   ↓ 5分钟后
半开状态（Half-Open）：放行少量请求测试
   ├── 成功 → 恢复正常状态
   └── 失败 → 继续熔断
```

**4. 限流**
- Gateway 层做全局限流（令牌桶算法）
- 热点接口做单独限流（如商品详情）

**5. 舱壁模式（Bulkhead）**
- 不同服务的调用使用独立线程池
- 一个服务故障只会耗尽自己的线程池，不影响其他服务

**6. 降级策略**（本项目可以这样设计）：
- 商品服务不可用时，订单服务：
  - 查询商品详情 → 返回缓存数据（降级读）
  - 扣减库存 → 先在 Redis 扣减，记录补偿消息
  - 返回"系统繁忙，请稍后重试"友好提示

**完整保护措施（建议）**：
```
Gateway 限流 → 服务超时控制 → Sentinel 熔断 → 降级响应
   ↓              ↓              ↓          ↓
防止流量洪峰   防止长时间等待   故障隔离   优雅失败
```

### 场景9：用户端三级缓存架构设计

**Q：从用户打开APP到看到商品列表，经过哪些缓存层级？**

**参考答案**：

**一级缓存：浏览器/APP 缓存**
- 静态资源（JS/CSS/图片）设置 HTTP 缓存头（Cache-Control、ETag）
- CDN 加速：商品图片、静态页面存 CDN
- 首次访问慢，但后续访问极快

**二级缓存：Nginx / CDN 缓存**
- 商品详情页静态化：商品介绍HTML片段缓存到 Nginx
- 缓存时间：商品详情1小时，热商品图片7天
- 商品上架/下架时主动刷新缓存

**三级缓存：本地缓存（JVM内，Caffeine）**
- 热点商品详情、分类列表等
- 缓存时间：5分钟（短有效期，自动过期）
- 优点：极快，无网络开销
- 缺点：多实例可能数据不一致，内存占用

**四级缓存：Redis 分布式缓存**
- 库存数据：`stock:sku:{skuId}`
- 商品基本信息：`product:info:{productId}`
- 用户信息、购物车：`user:cart:{userId}`
- 缓存时间：15分钟~1小时（根据数据实时性要求）

**数据库：MySQL**
- 最终数据来源
- 缓存未命中时查询数据库，然后回写缓存

**缓存穿透/击穿/雪崩防护**：
- 穿透：缓存空值 + 参数校验
- 击穿：热点 key 永不过期 + 互斥锁
- 雪崩：随机化过期时间 + 多级缓存 + 服务降级

### 场景10：消息队列消息积压问题

**Q：如果MQ消费者挂了一段时间，消息积压了几十万条，恢复后怎么处理？**

**参考答案**：

**1. 紧急处理**
- 扩容消费者实例数量（增加消费能力）
- 关闭非核心消费者，优先处理核心消息
- 临时增加消费者线程数

**2. 批量消费**
- 把单条消费改成批量消费（如一次拿100条，批量处理）
- 减少IO交互（批量写数据库、批量调用接口）

**3. 消息丢弃/降级**
- 非关键消息（如日志、通知）：如果时效性要求不高，可选择丢弃
- 记录到日志文件，后续离线处理

**4. 死信队列（DLQ）**
- 消费失败多次（如16次）后，自动转投到死信队列
- 人工处理死信队列中的消息
- 防止坏消息一直阻塞正常消费

**5. 预防措施**：
- 监控消息堆积量（监控 Topic 的 Message Delay）
- 设置告警阈值（如堆积超过1万条告警运维）
- 消费者健康检查，自动重启
- 消息 TTL 设置：防止消息永远堆积

**本项目中的消息监控**：
- RocketMQ Dashboard 可视化查看消息堆积
- 关键 Topic 设置消息超时时间

### 场景11：唯一ID生成方案

**Q：分布式环境下如何生成全局唯一的订单号？**

**参考答案**：

**常见方案对比**：

| 方案 | 原理 | 优点 | 缺点 |
|-----|------|------|------|
| **UUID** | 128位随机数 | 简单、无依赖 | 太长、无序、存储索引性能差 |
| **数据库自增ID** | MySQL AUTO_INCREMENT | 有序、简单 | 单点问题、性能瓶颈、分库分表时需改造 |
| **号段模式** | 每个服务从数据库取一批ID范围 | 性能好（一次取一批） | 依赖数据库，单点故障 |
| **Redis INCR** | Redis原子递增 | 高性能 | Redis持久化问题、宕机可能重复 |
| **雪花算法（Snowflake）** | 时间戳+机器ID+序列号 ⭐ | 64位long、有序、高性能、无单点 | 时钟回拨需要处理 |

**雪花算法结构（64位）**：
```
1位符号位（0） + 41位时间戳（毫秒级，可用69年）
+ 10位工作机器ID（最多1024个节点）
+ 12位序列号（每毫秒最多4096个ID）

例：0 | 0001100...0011011 | 0001100101 | 000000000001
    ↑          ↑              ↑              ↑
   正数      时间戳         机器ID         序列号
```

**本项目使用雪花算法**（`OrderNoGenerator`）：
- 每毫秒可以生成 4096 个订单号（足够应对大促）
- 订单号是递增的，数据库索引性能好
- 64位 long 类型，存储和传输效率高

**时钟回拨问题处理**：
- 简单方案：检测到回拨时，等待时钟追上（或直接报错）
- 复杂方案：使用之前的序列号继续 + 记录最大时间戳

**订单号建议格式**：
```
SH + 年月日(6位) + 雪花ID(至少13位)  
例：SH241201 + 1234567890123 → SH2412011234567890123
```

### 场景12：数据库分库分表

**Q：订单表数据量预计1亿条，单表查询太慢，如何设计分库分表？**

**参考答案**：

**分库分表（Sharding）**：

**1. 垂直切分**（按业务分库）：
```
shop数据库 → shop_user + shop_product + shop_order + shop_payment
```

**2. 水平切分**（按数据量分表）：
- **按用户ID取模**：`order_info_{userId % 32}` → 32张表
  - 优点：某个用户的订单都在同一张表，查询方便
  - 缺点：热点用户（如测试账号）可能导致某张表数据过多

- **按时间（年月）**：`order_info_202401`、`order_info_202402`...
  - 优点：历史数据可归档、冷数据可迁移
  - 缺点：跨月查询麻烦

- **组合方案**：先按用户ID分库，再按时间分表
  - `order_db_{userId % 4}.order_info_{yyyyMM}` → 4库 × N月表

**3. 工具选择**：
- **ShardingSphere-JDBC**（推荐）：Java 端分片，性能好，无运维成本
- **MyCat**：代理层分片，独立部署，跨语言

**4. 分片后查询挑战**：
- **只查自己的数据**：按用户ID查询 → 直接路由到目标表（快）
- **跨分片查询**：按商品ID查订单 → 需要扫描所有分片（慢）→ 用ES索引辅助查询
- **分页排序**：每个分片取 LIMIT M,N，在内存中合并排序 → 深分页性能差
- **分布式事务**：跨库事务需要 XA 或 Seata

**5. 扩容问题**：
- 32张表不够 → 扩展到64张 → 需要数据迁移（麻烦）
- 建议：一开始就预留足够的分片数（如一开始就分256张，大部分空着也不占多少空间）

**6. 本项目的简单建议**：
- 学习阶段可以先不分片，保持简单
- 用读写分离提升读性能（主库写，从库读）
- 历史订单做归档（如一年前的订单移到 archive 库）

**Q4：用户下单的完整链路，从前端到后端？**

   l. 发送 RocketMQ 延时消息（30分钟后检查是否支付）

4. 前端：展示订单 → 引导用户支付

6. shop-payment：发送 MQ 消息 topic_payment_callback

8. （30分钟后）如果订单仍"待付款"：

**Q5：怎么保证支付成功后订单状态一定能更新？**

**参考答案思路**：

**三步走，层层保障**：

1. **消息队列可靠投递**：支付服务发送 MQ 消息
   - 同步发送 + 失败重试（确保消息发到 MQ）
   - MQ Broker 持久化（消息存到磁盘）
   - 消费者手动 ACK（处理成功后才告诉 MQ，失败则重试）

2. **消费端幂等处理**：订单服务收到消息时
   - 先查订单状态 → 如果已经是"已支付"，直接返回成功（防止重复处理）
   - 状态机判断：只有"待付款"才能转为"已支付"
   - 乐观锁更新：`UPDATE order SET status = 2 WHERE id = ? AND status = 0`
   - 更新行数为0说明并发问题，记录日志即可

3. **补偿机制**（兜底）：
   - 定时任务定时扫描"待付款超过30分钟"和"支付状态不一致"的订单
   - 向支付服务主动查询支付结果
   - 发现不一致则修复（可能是支付成功但 MQ 消息丢失）
   - 人工对账：每天生成对账文件，订单系统和支付系统逐笔核对

**Q6：如果 Redis 挂了，库存扣减怎么办？**

**参考答案思路**：

**降级方案设计**：

1. **Redis 不可用检测**：
   - 连续 N 次 Redis 操作失败 → 标记为不可用状态
   - 定时心跳检测，恢复后自动切回 Redis 模式

2. **降级到数据库乐观锁**：
   ```java
   try {
       // 尝试 Redis 扣减
       redisDeductStock(skuId, quantity, orderNo);
   } catch (RedisException e) {
       // Redis 挂了，降级为数据库乐观锁
       int updated = productMapper.update(
           "UPDATE product_sku SET stock = stock - #{qty}, version = version + 1 
            WHERE id = #{skuId} AND stock >= #{qty}", params);
       if (updated == 0) throw new BusinessException("库存不足");
   }
   ```

3. **Redis 恢复后的数据同步**：
   - 不可用期间的库存变更都记录在数据库
   - Redis 恢复后：从数据库读取最新库存 → 写回 Redis（`SETNX` 不覆盖已有）

4. **Redis 高可用**（预防措施）：
   - Redis Sentinel 哨兵模式（监控+自动故障转移）
   - Redis Cluster 集群模式（数据分片，高可用+高性能）
   - 生产环境至少一主一从，避免单机故障

**Q7：Elasticsearch 和 MySQL 数据不一致怎么办？**

**参考答案思路**：

**1. 实时数据（价格、库存）不存 ES**
- 搜索结果只包含商品基本信息（名称、图片、分类）
- 价格和库存实时从 MySQL 查询合并（`ProductSearchServiceImpl.mergeRealtimeData`）
- 搜索响应时间 = ES 搜索时间 + 少量数据库查询时间（依然很快）

**2. 准实时数据（商品名称、描述）异步同步**
- 商品创建/更新 → 发 MQ 消息 → 消费者写 ES
- 延迟：几秒级别（用户体验可接受）

**3. 全量同步（兜底）**：
- 定时任务（如每天凌晨）全量扫描 MySQL → 重建 ES 索引
- 修复漏同步的数据

**4. 增量同步（Canal 监听 binlog，推荐）**：
- Canal 订阅 MySQL binlog
- 解析增删改事件 → 同步到 ES
- 零业务侵入，即使商品服务挂了也能同步

**5. 人工触发**：
- 管理后台提供"同步到ES"按钮
- 紧急情况管理员手动触发

**Q8：前端怎么防止重复提交订单？**

**参考答案思路**：

**1. 按钮防抖（前端第一道防线）**
- 点击后按钮立即置灰 + 显示 loading 动画
- 3秒内不允许再次点击
- 收到响应后恢复按钮状态

**2. 幂等Token机制（前后端配合）**
```
进入订单页 → 请求 GET /api/order/idempotent-token
              ↓
            后端生成UUID → 存入 Redis（5分钟有效）
              ↓
            前端拿到 Token
              ↓
提交订单时带上 Header: X-Idempotent-Token=xxxx
              ↓
后端 Redis Lua脚本: GETDEL key
   - 获取成功（第一次提交）→ 继续创建订单
   - 获取失败（Token已被删除）→ 返回"请勿重复提交"
```

**3. 请求ID + 后端去重**
- 前端每个请求生成唯一请求ID（如 timestamp + random）
- 后端用 Redis Set 检查请求ID是否已处理过
- 处理过的请求直接返回上次的处理结果

**4. 后端分布式锁**
- 同一用户同一时间只能创建一个订单
- `SET order:create:{userId} locked NX EX 30`

**5. 数据库唯一约束（最后一道防线）**
- 业务不适用订单号，但可以用"用户ID+商品ID+日期"的唯一约束（如秒杀场景）

**Q9：怎么设计一个秒杀系统？**

**参考答案思路（简要版）**：

**三大挑战**：高并发（10万QPS）、库存（不能超卖）、公平性（先来先得）

**架构设计**：

```
【接入层】Nginx/CDN/前端限流 + 按钮置灰
   ↓
【网关层】Gateway 限流（令牌桶，1万QPS）+ 黑名单拦截
   ↓
【业务层】微服务集群部署 + Sentinel 熔断
   ├── 秒杀资格校验（活动是否开始、用户是否已抢过）
   ├── 库存预扣（Redis Lua脚本，毫秒级）
   ├── 异步下单（发MQ消息，不等待结果）
   └── 立即返回"排队中"页面给用户
   ↓
【消息队列】RocketMQ（削峰）：10万QPS进来，1万QPS慢慢消费
   ↓
【数据层】消费者慢慢处理：创建订单 + 扣减数据库库存（最终一致）
   ↓
【通知层】WebSocket/轮询通知用户抢购结果
```

**核心技术点**：
1. **库存预热**：开始前把商品库存加载到 Redis
2. **库存分片**：把1000件库存拆成 `stock:sku:0~9` 10个 key，分散压力
3. **Lua脚本扣减**：`GET stock → DECRBY → SETNX 防止重复` 原子执行
4. **异步下单**：扣 Redis 成功就返回用户"成功"，后续异步创建订单
5. **一人一单**：Redis `SETNX user:seckill:{userId}:{activityId}`
6. **限流**：用户级限流 + IP级限流 + 接口级限流
7. **降级**：商品详情降级读缓存、推荐系统关闭
8. **最终一致性**：扣Redis成功但创建订单失败 → 发"回滚库存"消息 + 定时补偿

**Q10：如何设计 RBAC 权限系统？**

**参考答案思路（本项目 shop-admin）**：

**RBAC（Role-Based Access Control，基于角色的访问控制）**：

**核心模型（5张表）**：

```
admin_user（管理员用户表）
├── id, username, password, status...
│
admin_role（角色表）
├── id, name, code, description...
│
admin_permission（权限表，菜单+按钮）
├── id, parent_id, name, code, type(菜单/按钮), url, icon, sort...
│
admin_user_role（用户-角色关联表）
├── user_id, role_id  ← 一个用户可以有多个角色
│
admin_role_permission（角色-权限关联表）
├── role_id, permission_id  ← 一个角色可以有多个权限
```

**权限校验流程**：

```
用户登录 → 返回 Token + 用户信息 + 权限列表
   ↓
前端根据权限列表动态渲染菜单和按钮
   ↓
请求后端接口 → Gateway 鉴权 → 进入管理服务
   ↓
@RequirePermission("product:edit") 注解（AOP切面校验）
   ↓
切面检查用户角色对应的权限列表是否包含 product:edit
   ├── 是 → 执行业务逻辑
   └── 否 → 返回 403 无权限
```

**数据权限扩展**：
- 部门数据隔离：只能查看/操作本部门和子部门的数据
- 通过 AOP 切面自动在 SQL 上追加 `dept_id IN (用户部门列表)` 条件

**扩展到多租户（可选）**：
- 增加 tenant_id 字段
- 所有查询自动追加租户过滤
