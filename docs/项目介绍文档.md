# 项目介绍文档 - Shop 电商平台

> 适用人群：初级Java开发者 / 初学者
> 项目类型：微服务电商平台（学习项目）
> 技术栈：Spring Boot 4 + Spring Cloud 2025 + Spring Cloud Alibaba + MySQL + Redis + RocketMQ + Elasticsearch + MinIO

---

## 一、项目简介

本项目是一个**电商平台学习项目**，采用**微服务架构**，包含用户端、商家端和管理端三个前端应用，以及多个后端微服务。项目覆盖了电商系统的核心业务链路：用户注册登录 → 浏览商品 → 加入购物车 → 下单支付 → 物流跟踪 → 退款售后。

### 1.1 项目目标

- 学习微服务架构的设计与实现
- 理解分布式事务、消息队列、搜索引擎等中间件在实际业务中的使用
- 掌握 Spring Cloud Alibaba 全家桶的使用（Nacos、Seata、OpenFeign等）
- 了解电商系统的核心业务流程与技术解决方案

### 1.2 三大客户端

| 客户端 | 面向人群 | 核心功能 |
|--------|---------|---------|
| **用户端** | 普通消费者 | 注册/登录、商品浏览、搜索、购物车、下单、支付、评价 |
| **商家端** | 商家/卖家 | 开店申请、商品发布、库存管理、订单发货、结算 |
| **管理端** | 平台管理员 | 用户管理、商家审核、商品管理、订单管理、数据统计、权限管理 |

### 2.2 架构特点

1. **前后端分离**：后端提供REST API，前端使用Vue 3独立部署
2. **微服务架构**：按业务领域拆分成多个独立服务
3. **统一网关**：所有请求经过Gateway进行鉴权和路由
4. **中间件解耦**：消息队列处理异步任务，搜索引擎处理复杂查询

## 三、微服务模块详解

### 3.1 shop-gateway（网关服务）

**端口**：8844
**职责**：所有外部请求的统一入口

**核心功能**：
1. **路由转发**：根据请求路径转发到对应微服务
   - `/api/user/**` → shop-user
   - `/api/product/**` → shop-product
   - `/api/cart/**` → shop-cart
   - `/api/order/**` → shop-order
   - `/api/payment/**` → shop-payment
   - `/api/merchant/**` → shop-merchant
   - `/api/marketing/**` → shop-marketing
   - `/api/seckill/**` → shop-seckill
   - `/api/admin/**` → shop-admin
2. **权限校验**：使用Sa-Token校验登录状态
3. **Header传递**：解析Token后，将`userId`、角色信息放入请求头传递给下游服务
4. **白名单机制**：登录、注册、商品浏览等接口无需鉴权
5. **防伪造Header**：清除外部传入的敏感Header，由网关统一写入

**关键代码位置**：
- 鉴权过滤器：[AuthGlobalFilter.java](file:///d:/workspace/business/Shop/shop-gateway/src/main/java/com/shop/gateway/filter/AuthGlobalFilter.java)
- 路由配置：[application.yml](file:///d:/workspace/business/Shop/shop-gateway/src/main/resources/application.yml)

## 五、中间件使用详解

### 5.1 Nacos（服务注册发现 + 配置中心）

**Nacos控制台**：默认地址 http://localhost:8848/nacos

**使用场景**：
1. **服务注册发现**：每个微服务启动时向Nacos注册自己的地址，其他服务通过Nacos发现它
2. **配置管理**：通过Nacos统一管理各服务的配置文件（如数据库连接、Redis地址等）
   - 公共配置：`common.yml`（所有服务共享）
   - 服务配置：`shop-user.yml`、`shop-product.yml`、`shop-order.yml`、`shop-marketing.yml`、`shop-seckill.yml` 等
   - 配置热更新：修改Nacos配置后，服务无需重启即可生效

**关键配置文件**：
- [Nacos配置脚本目录](file:///d:/workspace/business/Shop/docker/nacos/config/)

### 5.2 Redis（缓存 + 分布式锁）

**Redis用途**：

| 功能 | Key格式 | 说明 |
|------|---------|------|
| **Sa-Token会话** | `satoken:login:token:{token}` | 存储用户登录状态 |
| **库存缓存** | `stock:sku:{skuId}` | SKU可用库存，Lua脚本扣减 |
| **库存扣减记录** | `stock:deduct:{skuId}` | Set结构，存已扣减的订单号 |
| **库存回滚记录** | `stock:rollback:{skuId}` | Set结构，存已回滚的订单号 |
| **热门搜索词** | `product:hot_keywords` | ZSet结构，存搜索词+热度分数 |
| **下单分布式锁** | `order:create:{userId}` | 防止同一用户并发下单 |
| **幂等Token** | `order:idempotent:{token}` | 防止用户重复提交订单 |

**核心亮点 - Redis + Lua脚本扣库存**（在`StockServiceImpl.java`中）：
- Lua脚本在Redis中**原子执行**，天然防止并发问题
- 比数据库乐观锁性能更高（减少数据库压力）
- 支持幂等：同一订单号不会重复扣减

### 5.3 RocketMQ（消息队列）

**使用场景**：

| Topic | 生产者 | 消费者 | 用途 |
|-------|--------|--------|------|
| `topic_order_timeout` | shop-order | shop-order | 订单超时自动取消（30分钟延时消息） |
| `topic_payment_callback` | shop-payment | shop-order | 支付成功通知订单服务更新状态 |
| `product_sync_topic` | shop-product | shop-product | 商品变更同步到Elasticsearch |
| `topic_seckill_order` | shop-seckill(或shop-order) | shop-order | 秒杀抢购成功后异步创建订单 |

**消息队列的好处（小白理解）**：
1. **异步解耦**：支付服务不用等订单服务处理完，发完消息就可以返回给用户
2. **削峰填谷**：大促时大量支付回调不会把订单服务压垮
3. **可靠投递**：即使订单服务暂时挂了，消息不会丢，等服务恢复后继续消费

### 5.4 Elasticsearch（搜索引擎）

**索引**：`product`（商品索引）

**核心功能**：
1. **全文检索**：ik中文分词，支持按商品名称/副标题搜索
2. **筛选**：分类、品牌、价格区间
3. **排序**：综合排序、价格升降序
4. **高亮**：搜索结果中关键词高亮显示（`<em>关键词</em>`）
5. **聚合**：统计匹配结果中各品牌/分类的商品数量
6. **搜索建议**：输入时实时推荐商品名

**ES vs MySQL搜索对比**：
- MySQL的`LIKE '%关键词%'`无法使用索引，全表扫描极慢
- ES使用倒排索引，搜索速度快几个数量级
- ES支持中文分词，搜索更智能（搜"手机"也能找到"智能手机"）

- 商品同步消费者：[ProductSyncConsumer.java](file:///d:/workspace/business/Shop/shop-product/src/main/java/com/shop/product/consumer/ProductSyncConsumer.java)

## 六、核心业务流程

### 6.1 用户下单全流程

```
用户浏览商品 → 加入购物车 → 去结算 → 提交订单 → 支付 → 等待发货 → 收货确认
     ↓              ↓          ↓         ↓         ↓           ↓          ↓
  ES搜索       Redis缓存   校验库存   ①创建订单  ②支付回调   ③发货      确认收货
                          扣减库存   (Seata     (RocketMQ)    (商家)      (更新状态)
                          分布式事务)
                          ↓
                      ④发送延时消息(30分钟)
                          ↓
                      超时未支付 → 自动取消 → 回滚库存
```

**下单步骤详解**（`OrderServiceImpl.createOrder`）：
1. **幂等校验**：检查前端传递的幂等Token是否已使用
2. **获取分布式锁**：同一用户不能同时创建多个订单
3. **Feign获取商品信息**：校验SKU是否存在、库存是否充足
4. **Feign获取地址信息**：校验收货地址
5. **生成订单号**：雪花算法生成全局唯一订单号
6. **计算金额 + 创建订单**：插入order_info、order_item、order_address
7. **Feign调用扣减库存**：通过Seata分布式事务保证一致性
8. **发送RocketMQ延时消息**：30分钟后检查是否支付
9. **删除购物车中已下单商品**（失败不影响主流程，记录日志即可）
10. **释放分布式锁**

### 6.2 支付流程

```
用户支付 → 第三方支付平台 → 支付回调 → shop-payment处理 → 发送MQ消息 → shop-order消费
                ↓                                                     ↓
         返回支付结果                                            更新订单状态为"已支付"
```

### 6.3 商品搜索流程

```
用户输入关键词 → Gateway → shop-product → ES搜索（分词+筛选+排序+高亮+聚合）→ 合并实时价格/库存 → 返回结果
```

## 七、项目亮点

### 1. 分布式事务（Seata）
- 下单扣库存涉及跨服务数据一致性
- 使用Seata AT模式，自动生成回滚日志

### 2. 高性能库存扣减（Redis + Lua）
- Redis内存操作，速度极快
- Lua脚本原子执行，天然防并发
- 幂等性设计，防止重复扣减

### 3. 超时订单自动取消（RocketMQ延时消息）
- 下单时发送延时消息（等级16 = 30分钟）
- 消息消费时检查订单是否已支付
- 未支付则自动取消并回滚库存

### 4. 全文检索（Elasticsearch）
- ik中文分词
- 多维度筛选 + 排序 + 高亮 + 聚合
- 搜索建议 + 热门搜索词

### 5. 统一网关鉴权
- 所有请求经过Gateway
- Header防伪造
- Token自动续期
- 白名单机制

### 6. 完整的权限体系（管理端RBAC）
- 用户-角色-权限三层结构
- 数据权限控制（部门数据隔离）
- 操作日志记录

### 7. 订单状态机
- 严格的状态流转校验
- 防止非法操作（如未付款订单不能发货）
- 状态变更全记录（order_log）

### 8. 数据快照机制
- 地址快照：下单时复制地址，防止用户修改影响历史订单
- 价格快照：SKU价格随订单保存，不受后续调价影响

### 9. 乐观锁防并发
- 账户表：用户余额/积分修改
- SKU表：库存修改（降级方案）
- 订单状态更新

### 10. Nacos配置中心
- 配置集中管理
- 支持多环境配置
- 配置热更新（无需重启服务）

## 八、项目可改进/不足之处

### 8.1 功能不足

1. **支付方式单一**：目前只有支付流程框架，未对接真实的支付宝/微信支付
2. **物流对接**：未对接真实物流API，物流信息需手动录入
3. **推荐系统**：目前只有浏览足迹数据，未实现个性化推荐
4. **用户评价体系**：评价功能简单，缺少追评、晒单等
5. **商家结算系统**：结算功能不完善，缺少对账、提现等
6. **后台数据统计**：Dashboard仅有基础数据，缺少可视化图表
7. **消息通知**：下单、发货等状态变更未实现站内信/短信通知
8. **文件上传前端**：MinIO后端已就绪，但前端上传组件可能不完善

### 8.2 架构/技术可优化

1. **服务熔断降级**：未集成Sentinel，服务故障时无熔断保护
2. **链路追踪**：未集成SkyWalking/Zipkin，定位问题依赖日志
3. **分布式ID**：订单号使用雪花算法，但workerId未做服务区分
4. **数据库分库分表**：数据量大时需要考虑ShardingSphere
5. **二级缓存**：Caffeine + Redis 多级缓存可进一步提升性能
6. **缓存预热**：系统启动时未进行缓存预热
7. **缓存击穿/穿透**：热门商品缓存失效时可能打垮数据库
8. **接口幂等性**：部分接口缺少完善的幂等设计
9. **限流**：缺少接口限流保护（Gateway有基础框架但未完善）
10. **日志收集**：ELK/EFK日志系统未集成
11. **监控告警**：Prometheus + Grafana监控未集成
12. **容器化部署**：Docker Compose可运行但缺少K8s部署方案

### 8.3 安全可加强

1. **敏感数据加密**：手机号等只有框架，实际加密逻辑需完善
2. **XSS/CSRF防护**：前端和后端都需进一步加固
3. **SQL注入防护**：MyBatis-Plus已防SQL注入，但手写SQL需审查
4. **接口防刷**：缺少基于IP/用户的接口防刷机制
5. **密码策略**：缺少密码强度校验、定期修改等

### 8.4 代码/工程

1. **单元测试**：项目缺少单元测试覆盖
2. **API文档**：SpringDoc配置存在但可能不完善
3. **代码注释**：部分模块注释较少
4. **异常处理**：全局异常处理需要统一规范
5. **DTO/VO划分**：部分模块DTO/VO划分可更清晰
